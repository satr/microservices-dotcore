# Library Microservices (.NET)

A sample microservices solution for a library borrowing workflow, built entirely on .NET 9.
Generated by an agent.

## Architecture

```
┌────────────────────────────────────────────────────────────────────┐
│ Browser                                                            │
│  └── frontend (Razor Pages, port 8080)                            │
│        ├── GET  /api/users/by-name/{name}  → users-service        │
│        ├── GET  /api/books/search?query=…  → books-service        │
│        └── POST /api/cart/items            → booking-service      │
│                                                                    │
│ booking-service ──publish──▶ RabbitMQ ◀──subscribe── workflow-saga│
│ workflow-saga   ──publish──▶ RabbitMQ ◀──subscribe── booking-service│
└────────────────────────────────────────────────────────────────────┘
```

### Services

| Service | Folder | Port |
|---|---|---|
| **users-service** | `services/UsersService` | 5001 |
| **books-service** | `services/BooksService` | 5002 |
| **booking-service** | `services/BookingService` | 5003 |
| **workflow-saga** | `workflow-saga/WorkflowSaga` | — |
| **frontend** | `frontend/LibraryWeb` | 8080 |
| **RabbitMQ** | management UI at [localhost:15672](http://localhost:15672) | 5672 |
| **Kafka (Stage 3 scaffold)** | broker `kafka` + UI at [localhost:8081](http://localhost:8081) | 29092 |
| **PostgreSQL** | `postgres` container (unused by default) | 5432 |

### Seeded data

- Users: `user1`, `user2`
- Books: `Book1 / Author1`, `Book2 / Author2`, `Book3 / Author3`

## Quick Start

```bash
make up        # build images and start all containers
```

Open [http://localhost:8080](http://localhost:8080).

## Common Make commands

```bash
make up              # build + start everything
make down            # stop containers
make restart         # full rebuild and restart
make logs            # tail all logs
make ps              # show container status
make clean           # stop + delete volumes

# Rebuild a single service (no downtime for others)
make build-users
make build-books
make build-booking
make build-saga
make build-frontend

# Per-service log tailing
make logs-frontend
make logs-booking
make logs-saga
```

### Optional Kafka scaffold (Stage 3)

Kafka is added as an optional local stack and does not replace RabbitMQ yet.
The UI container uses the actively maintained `kafbat/kafka-ui` fork.
Use `localhost:29092` from the host; `kafka:9092` is reserved for Docker-internal clients such as `kafka-ui`.

```bash
make kafka-up
make kafka-ps
make kafka-logs
make kafka-down
```

Kafka UI: [http://localhost:8081](http://localhost:8081)

## Workflow (MassTransit Saga)

When a user hits **Add to Cart**, **Remove**, or **Complete Borrowing**:

1. `booking-service` publishes a request message (`AddToCartRequested` / `RemoveFromCartRequested` / `CompleteBorrowingRequested`) to RabbitMQ.
2. `workflow-saga` consumes the message via `BorrowingStateMachine`, applies business logic, then publishes a result event (`CartItemAdded` / `CartItemRemoved` / `BorrowingCompleted`).
3. `booking-service` consumers apply the result event to the in-memory `ICartRepository`.

## Data Layer

Each service's repository is behind an interface (`ICartRepository`, `IBookRepository`, `IUserRepository`). The default implementation is in-memory. `PostgresCartRepository` (stub) is included in `BookingService` as the drop-in replacement once the `postgres` container is wired up.
