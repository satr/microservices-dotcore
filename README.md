# Library Microservices (.NET)

A sample microservices solution for a library borrowing workflow, built entirely on .NET 9.
Generated by an agent.

## Stages

- **[Stage 1](https://github.com/satr/microservices-dotcore/tree/stage1)** — [details](https://github.com/satr/microservices-dotcore/blob/main/Stage1.md)
  - Three microservices (users, books, booking), Razor Pages frontend
  - RabbitMQ messaging, MassTransit saga (`BorrowingStateMachine`)
  - In-memory storage, Docker Compose

- **[Stage 2](https://github.com/satr/microservices-dotcore/tree/stage2)** — [details](https://github.com/satr/microservices-dotcore/blob/main/Stage2.md)
  - PostgreSQL persistence (EF Core, one DB per service)
  - Polly retry / circuit-breaker resilience
  - OpenTelemetry tracing exported to Jaeger
  - Book inventory stock tracking

- **[Stage 3](https://github.com/satr/microservices-dotcore/tree/stage3)** — [details](https://github.com/satr/microservices-dotcore/blob/main/Stage3.md)
  - Kafka transport (opt-in via `Messaging:Provider=Kafka` config)
  - Dual RabbitMQ / Kafka support — no breaking change
  - `kafbat/kafka-ui` (actively maintained fork)
  - Confluent Schema Registry container scaffold

- **[Stage 4](https://github.com/satr/microservices-dotcore/tree/stage4)** — [details](https://github.com/satr/microservices-dotcore/blob/main/Stage4.md)
  - URL-segment API versioning (`/api/v1/…`) on all services
  - `MessageEnvelope<T>` for Kafka message schema versioning
  - Protobuf `.proto` definitions for all message contracts

## Make Commands

### Core stack

```bash
make up              # build images + start all containers
make down            # stop and remove containers
make restart         # down + up
make logs            # tail logs for all services
make ps              # show container status
make clean           # stop + delete volumes (resets all DBs)
```

### Rebuild a single service

```bash
make build-users
make build-books
make build-booking
make build-saga
make build-frontend
```

### Per-service log tailing

```bash
make logs-users
make logs-books
make logs-booking
make logs-saga
make logs-frontend
```

### Kafka (opt-in)

Starts Kafka, Schema Registry and Kafka UI alongside the default RabbitMQ stack.
Services switch to Kafka when `Messaging__Provider=Kafka` is set (already configured in `docker-compose.kafka.yml`).

```bash
make kafka-up              # start Kafka + Schema Registry + Kafka UI only
make kafka-stack-up        # build + start full stack in Kafka mode
make kafka-stack-restart   # rebuild booking-service + workflow-saga in Kafka mode
make kafka-ps              # show Kafka container status
make kafka-logs            # tail Kafka + Schema Registry + Kafka UI logs
make kafka-down            # stop and remove Kafka containers
```

### Schema Registry

```bash
make schema-check               # list registered subjects + global compatibility mode
make schema-set-full-transitive # enforce FULL_TRANSITIVE compatibility (blocks breaking changes)
```

## Borrowing Workflow (MassTransit Saga)

When a user hits **Add to Cart**, **Remove**, or **Complete Borrowing**:

1. `booking-service` publishes a command (`AddToCartRequested` / `RemoveFromCartRequested` / `CompleteBorrowingRequested`) to RabbitMQ or Kafka.
2. `workflow-saga` (`BorrowingStateMachine`) receives the command and publishes an outcome event (`CartItemAdded` / `CartItemRemoved` / `BorrowingCompleted` / `AddToCartFailed`).
3. `booking-service` consumers handle the outcome:
   - `CartItemAdded` → deduct stock in `BookInventories`, add row to `CartItems`; if stock is 0, publish `AddToCartFailed` instead
   - `CartItemRemoved` → remove row from `CartItems`, restore stock in `BookInventories`
   - `BorrowingCompleted` → clear all `CartItems` for the user
   - `AddToCartFailed` → record row in `CartItemFailures` (shown as dismissible alerts in the UI)

## Data Layer

Each service uses its own PostgreSQL database (EF Core + migrations applied on startup):

| Database | Service | Key tables |
|---|---|---|
| `users_db` | users-service | `Users` |
| `books_db` | books-service | `Books` |
| `booking_db` | booking-service | `CartItems`, `CartItemFailures`, `BookInventories` |

Repositories are behind interfaces (`ICartRepository`, `IBookRepository`, `IUserRepository`) — easy to swap implementations.

## API Versioning

All REST endpoints use URL-segment versioning. Current version: **v1**.

```
GET  /api/v1/books/search?query=…
GET  /api/v1/books/{id}
GET  /api/v1/users/by-name/{name}
GET  /api/v1/cart/{userId}
POST /api/v1/cart/items
DELETE /api/v1/cart/items/{bookId}?userId=…
POST /api/v1/cart/complete?userId=…
GET  /api/v1/cart/failures/{userId}
DELETE /api/v1/cart/failures/{userId}/{bookId}
GET  /api/v1/inventory/stock/{bookId}
POST /api/v1/inventory/stock/batch
```

Every response includes the header `api-supported-versions: 1.0`.
